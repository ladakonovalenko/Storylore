## ⚠️ Важливий момент щодо API

У наданій специфікації для Таймлайну задокументовано **лише**:
```
POST /characters/events – створити подію
```
GET-ендпоінт для читання списку подій не вказаний. Без нього сторінка не може
показати вже існуючі події після перезавантаження.

Щоб модуль працював вже зараз, `getCharacterEvents()` в `api/timeline.js`
пробує три варіанти по черзі:
1. `GET /characters/{id}/events`
2. `GET /characters/events?character_id=…`
3. `GET /characters/{id}` — якщо події вкладені в об'єкт персонажа як `events` чи `timeline`

Якщо жоден не відповість — функція повертає `null`, і сторінка показує
жовте попередження користувачу, що дані не персистентні між сесіями (це
обмеження бекенду, не фронтенду). Нові події, додані в поточному сеансі,
показуються оптимістично (без перезапиту).

**Коли матимеш точний GET-ендпоінт** — заміни в `api/timeline.js` функцію
`getCharacterEvents` на прямий виклик; решта компонентів (`TimelinePage`,
`TimelineEntry`) не зміняться, бо отримують вже готовий масив подій.

## Поля події (припущення, бо Pydantic-модель невідома)

Форма надсилає в `POST /characters/events`:
```
{
  character_id,
  title,        // назва події, обов'язково
  date_label,   // довільний текст ("Третій рік Великої Війни"), обов'язково
  sort_order,   // число для сортування на клієнті, необов'язково
  event_type,   // довільний текст з підказками (battle, betrayal…), необов'язково
  description,  // необов'язково
}
```
Якщо реальна модель відрізняється — поля легко перейменувати в одному місці
(`EventForm.jsx`, об'єкт у `handleSubmit`).

## Реалізовано

- Вибір персонажа (фільтрується активним проєктом, як і в Модулі 4)
- Вертикальна стрічка подій із конекторною лінією та іконкою за типом події
- Сортування за `sort_order`
- Форма створення з валідацією (назва + дата обов'язкові)
- Порожній стан, стан завантаження, toast про успіх/помилку
- Попередження про обмеження API (відсутність GET) винесене прямо в інтерфейс,
  а не лише в код
